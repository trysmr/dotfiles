# Release PR Creation Workflow

このスキルは`staging`ブランチを`main`ブランチへマージする本番リリース用PRを作成します。
通常のPR作成は`pr`スキルを使用してください。

**前提条件**:
- `staging`と`main`の両ブランチが存在すること
- stagingに含まれる各PRが既にレビュー・テスト・staging検証完了していること

## prスキルとの差分

| 項目 | prスキル | pr-releaseスキル |
|------|---------|-----------------|
| ベース/ソース | プロジェクトルールに従う | `main`<-`staging`固定 |
| Test・Linter | 必須 | 各PRで実施済みなので省略 |
| Pre-PR Review | 推奨 | 各PRでレビュー済みなので省略 |
| 本文テンプレート | 単一機能向け | 複数PR集約向け |
| タイトル形式 | 自由 | `[Release] staging -> main (YYYY-MM-DD): 要約`固定 |

## 実行手順

### 1. ブランチ確認

```bash
git fetch origin
git branch -a
```

`main`と`staging`の両方が存在することを確認します。

### 2. 含まれるPRの抽出

`main`から`staging`への差分に含まれるマージ済みPRを抽出します:

```bash
# コミット履歴からPR番号を抽出
git log origin/main..origin/staging --oneline | grep -oE '#[0-9]+' | sort -u
```

各PRの詳細を確認:

```bash
gh pr view <番号> --json number,title,body
```

抽出したPRの情報を整理:
- PR番号
- タイトル
- 主要な変更点（箇条書き2-4項目）

### 3. PR情報の準備

以下の情報を準備してユーザーに提示します。

- **ベースブランチ**: `main`（固定）
- **ソースブランチ**: `staging`（固定）
- **PRタイトル**: 後述のフォーマットに従う
- **PR本文**: 後述のテンプレートに従う

#### タイトルの書き方

```
[Release] staging -> main (YYYY-MM-DD): 主要変更の要約
```

- 日付はリリース予定日（または作成日）
- 主要変更の要約は1-2項目を**普通形・体言止め**で簡潔に（prスキルの「タイトルの書き方」に準拠）
- 例: `[Release] staging -> main (YYYY-MM-DD): インフラ移行、ライブラリアップデート`

#### PR本文テンプレート

```markdown
## 概要
ステージング環境で検証完了した[主要機能の概要]を本番環境にリリースします。

## 変更点
### PR1のタイトル（#PR番号1）
- 変更内容の要約1
- 変更内容の要約2

### PR2のタイトル（#PR番号2）
- 変更内容の要約1

## テスト計画
- [x] RuboCop静的解析パス
- [x] stylelint静的解析パス
- [x] 全テストN件パス(0 failures, 0 errors)
- [x] JSテストパス
- [x] staging環境で動作確認
- [ ] 本番デプロイ後の確認
  - [ ] アプリケーションが正常に起動することを確認
  - [ ] 主要機能が正常に動作することを確認
  - [ ] 変更内容に固有の確認項目
```

#### 概要の書き方

定型文「ステージング環境で検証完了した〜を本番環境にリリースします。」を使う。
含まれる主要機能を1文で要約する。

#### 変更点の書き方

- 各PRを`### PRタイトル（#PR番号）`のサブセクションで列挙
- PR番号は`#NNN`形式（GitHubが自動リンク化）
- 各PRに2-4項目の箇条書きで主要変更を記載
- 複数PRが同じ機能領域にまたがる場合は機能領域でグループ化してもよい

#### テスト計画の書き方

prスキルのテスト計画書式に「本番デプロイ後の確認」を追加した形で記載する。

以下の順序で記載する:

1. **RuboCop**: `[x] RuboCop静的解析パス`
2. **テスト結果**: `[x] [スコープ]N件パス(0 failures, 0 errors)`
   - 全テスト実行時: `全テストN件パス(0 failures, 0 errors)`
   - 特定テスト実行時: `コントローラテストN件パス(0 failures, 0 errors)`
3. **ステージング確認**: `[x] staging環境で動作確認`
4. **本番デプロイ後の確認**（リリース固有・ネスト箇条書き）
   - `[ ] アプリケーションが正常に起動することを確認`
   - `[ ] 主要機能が正常に動作することを確認`
   - `[ ] 変更内容に固有の確認項目`（外部API連携、ヘルスチェック、認証フロー等）

リリース前は`[ ]`、デプロイ後に`[x]`に更新する運用。1-3は各PRで検証済みなので`[x]`、本番デプロイ後の確認のみ`[ ]`で開始する。

### 4. ユーザー確認

PR情報（タイトル・本文）をユーザーに提示し、「はい」「OK」「作成して」などの許可を得てください。

**重要**: ユーザーの許可なしにPRを作成しないでください。

### 5. PR作成

ユーザーの許可を得てから実行:

```bash
gh pr create --base main --head staging --title "タイトル" --body "$(cat <<'EOF'
## 概要
ステージング環境で検証完了した...を本番環境にリリースします。

## 変更点
### XxxX（#NNN）
- ...

## テスト計画
- [x] RuboCop静的解析パス
- [x] stylelint静的解析パス
- [x] 全テストN件パス(0 failures, 0 errors)
- [x] JSテストパス
- [x] staging環境で動作確認
- [ ] 本番デプロイ後の確認
  - [ ] アプリケーションが正常に起動することを確認
  - [ ] 主要機能が正常に動作することを確認
  - [ ] 変更内容に固有の確認項目
EOF
)"
```

### 6. PR URLの確認

PR作成後、URLをユーザーに提示します。

## チェックリスト

実行前に確認すべき項目:

- [ ] `git fetch origin`で最新のリモート情報を取得した
- [ ] `git branch -a`で`main`と`staging`の存在を確認した
- [ ] `main..staging`の差分から含まれるPRを抽出した
- [ ] 各PRの内容を確認し、本文用のサマリーを作成した
- [ ] タイトルが`[Release] staging -> main (YYYY-MM-DD): ...`形式
- [ ] 本文に「概要」「変更点」「テスト計画」を含めた
- [ ] 各変更点が`### タイトル（#番号）`形式で記載されている
- [ ] PR情報をユーザーに提示し、許可を得た

## Troubleshooting

| エラー | 原因 | 対処 |
|--------|------|------|
| `staging`または`main`が存在しない | プロジェクトでstagingブランチを使っていない | リリースPRは不要。通常の`pr`スキルを使う |
| `git log main..staging`が空 | stagingにマージ済みの変更がmainにない状態 | リリース対象がない。マージ済みPRを確認 |
| `gh pr create`が「a pull request for branch already exists」 | 既存のリリースPRがある | `gh pr view`で既存PRを確認し、必要なら更新 |
| `gh pr create`がTLS/証明書エラーで失敗 | サンドボックス環境のネットワーク制限 | `dangerouslyDisableSandbox: true`を指定して再実行する |
