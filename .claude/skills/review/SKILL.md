# review: 統合レビューオーケストレーター

change-reviewerとsecurity-reviewerを組み合わせ、変更差分を包括的にレビューする。

## 実行手順

### 1. レビュー対象の特定

引数やユーザーの指示に応じて、レビュー対象を決定する:

- **引数なし（デフォルト）**: ステージ済み + 未ステージの変更（`git diff HEAD`）
- **`--staged`**: ステージ済みの変更のみ（`git diff --cached`）
- **`--branch`**: 現在のブランチの全コミット（ベースブランチからの差分）
- **`--commit <hash>`**: 特定コミットの変更（`git show <hash>`）

```bash
# ブランチレビューの場合、ベースブランチを検出
git branch -a
# branch-strategy.mdのルールに従いベースブランチを決定
Read .claude/skills/_shared/branch-strategy.md
# 差分を取得
git diff <base>...HEAD
```

### 2. 変更ファイルの分類

変更されたファイルを以下のカテゴリに分類する:

- **セキュリティ関連**: 認証/認可、入力処理、暗号化、権限管理、環境変数、API エンドポイント
- **一般コード**: 上記以外のすべてのコード変更

### 3. レビューの実行

#### 3a. 品質・保守性レビュー（常に実行）

change-reviewer Agentに以下を委譲:
- 変更差分の全文
- レビュー観点: パフォーマンス（N+1、ループ内処理）、保守性（命名、関数サイズ、テスト）、設計（結合度、パターン一貫性）

#### 3b. セキュリティレビュー（セキュリティ関連ファイルがある場合）

security-reviewer Agentに以下を委譲:
- セキュリティ関連ファイルの変更差分
- レビュー観点: 機密情報、インジェクション、認証・認可、入力バリデーション、情報漏洩

### 4. 結果の統合と報告

両エージェントの結果を統合し、重大度順に報告する:

```markdown
# レビュー結果

## 対象
- モード: branch（feature/add-auth → main）
- 変更ファイル数: 5
- セキュリティ関連: 2ファイル

## Critical / High
- [重大度] [カテゴリ] file:line — 問題の要約
  修正案: ...

## Medium / Low
- [重大度] [カテゴリ] file:line — 問題の要約
  修正案: ...

## 良い点
- [該当箇所]: 評価ポイント

## 総合評価
[PR作成可否の判断と、必要なアクション]
```

### 5. PR作成可否の判断

- **Critical/High指摘あり**: 「修正後に再レビューを推奨します」と報告
- **Medium以下のみ**: 「PR作成可能です。指摘事項は任意で対応してください」と報告
- **指摘なし**: 「問題は検出されませんでした。PR作成を推奨します」と報告

## チェックリスト

- [ ] レビュー対象を正しく特定した
- [ ] change-reviewerによる品質レビューを実行した
- [ ] セキュリティ関連ファイルがある場合、security-reviewerを実行した
- [ ] 結果を重大度順に統合して報告した
- [ ] PR作成可否の判断を提示した
